# 2026-06-27 Relay 诊断日志修复

## 问题
用户反馈"报错修复了，但是账号模型还是不行，而且日志都没了"。Console 只有 `[init] start` / `[init] done` / Security Warning，没有 `[relay-diagnostics]` 输出。

## 根因
`renderer.js` 是 ESM 环境（contextIsolation 开启，nodeIntegration 关闭），但 `fetchRelayDiagnostics` 函数用了 `require('path'/'os'/'fs')`（第 258-260 行）来读 runner-config.json。虽然这段代码被 try-catch 包裹（require 的 ReferenceError 被吞掉），但整个诊断流程依赖 renderer 自己探测 300+ 个端口 + HTTP fetch runner 的 `/__cursorpool__/diagnostics` 端点，非常脆弱。加上外层 `.catch(() => {})` 静默吞掉所有错误，用户看不到任何诊断输出。

## 修复方案
**架构调整**：让 main 进程代为请求 runner 内部状态，renderer 只调 IPC。

### 1. `js/utils/cursor-relay-proxy.js`
- 顶部新增 `const http = require('http');`
- 新增 `fetchRunnerInternals(port, timeoutMs=2000)` 辅助函数：用 http.get 请求 runner 的 `/__cursorpool__/diagnostics` 端点
- `buildRelayDiagnostics()` 在 return 之前调用 `fetchRunnerInternals(status.runner?.port)`，把结果合并到返回对象的 `runnerInternals` 字段
- 返回对象新增 3 个字段：`runnerInternals`（runner 进程内部 6 大类状态）、`runnerPort`、`runnerRunning`

### 2. `renderer.js`
- 完整重写 `fetchRelayDiagnostics()`：
  - 删掉所有 `require('path'/'os'/'fs')` 和 300+ 端口探测代码
  - 只调 `window.electronBridge.cursorRelayDiagnose()` IPC
  - 加 8 秒超时（Promise.race + setTimeout reject）
  - 函数入口有 `console.log` 确认被调用
  - 从 `diag.runnerInternals` 读取 runner 内部状态（proto root / auth 拦截 / 模型注入 / state guard / profile store / mode registry）
  - 新增 main 视角问题清单打印（`diag.issues`）
- `init()` 里的 `.catch(() => {})` 改为 `.catch((e) => console.warn('[relay-diagnostics] init 触发失败:', e.message))`
- 暴露 `window.__relayDiag = fetchRelayDiagnostics`，用户可在 DevTools Console 手动输入 `__relayDiag()` 触发
