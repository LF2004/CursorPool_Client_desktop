## 修复 agent 对话提前结束/主动结束问题（v2：纯硬信号方案）

### 设计原则（用户明确要求）
**不可用中英文关键词正则猜测模型意图**——日韩俄等语言会漏，且"我先读"也可能
是已完成动作描述。判定只能基于工具帧/状态机落地的事实：
- `toolCalls` 数组（本轮 pending 工具帧）
- `toolResultSummaries`（本轮实际执行过的工具）
- `session.todos`（结构化 todo 状态）
- `lastStreamSawDone` / `lastStreamStopReason`（流协议层完成信号）
- HTTP 状态码 / Node errno 码（协议级错误标识）

### 背景
用户反馈 agent 对话任务没完成就结束，多轮失败。分析会话历史
`C:\Users\Administrator\.cursorpool\relay\history\d245fbd6-37c3-4127-9ead-848d00bda8c8`
中三个失败请求（5989b585、ecd2955a、650cd896）后，定位到三种失败模式：

1. **5989b585**：`fetch failed`（网络异常抛出）→ 走 `failAgentRelaySession` → 直接标 `failed`，
   完全没经过未完成判定。
2. **ecd2955a**：模型说 "Let me read the code more carefully to understand the full flow."
   后流就结束了（没发工具调用），`looksLikeIncompleteContinuationText` 只匹配 placeholder 文本
   → 判定完成 → 标 `completed`，对话结束。
3. **650cd896**：agent 做了 4 轮"读代码→打补丁→补丁没命中→再读→再改"，每轮都说
   "现在我直接把这段改成…再做诊断"，最后撞上 HTTP 502 → `hasIncompleteWorkAtEndByMode`
   因 `!upstreamError` 守卫直接返回 false → 标 `failed`，4 轮工作全部丢失。

### 根因
- `looksLikeIncompleteContinuationText`（cursor-relay-runner.js）形同虚设，只匹配
  "Called N tool(s)" placeholder，不识别"接下来/现在直接/我先读/Let me read"等继续意图。
- `hasIncompleteWorkAtEndByMode` 默认逻辑和 plan-mode 实现都有 `!upstreamError` /
  `if (upstreamError) return false` 守卫，上游一报错就跳过未完成判定，直接走 failed。
- `isRecoverableStreamError` 只匹配 timeout 类，不覆盖 HTTP 502/503/504、fetch failed、
  连接重置等瞬时网关/网络错误。
- 主循环"可恢复错误"分支只调 `rememberUnfinishedAgentTask` 却没设
  `unfinishedWorkAtEnd = true`，导致历史状态仍是 failed。
- `failAgentRelaySession` 无条件标 failed，不检测本轮是否已有工作进行中。

### 修复（v2：纯硬信号，6 处，2 个文件）
1. **cursor-relay-runner.js `looksLikeIncompleteContinuationText`**：撤掉所有关键词
   正则，回归只匹配 relay 自己生成的 placeholder 协议串（"Called N tool(s)"）。
   新增 `streamEndedWithoutReliableCompletion(options)`：当 `!lastStreamSawDone` 且
   `stopReason ∈ {stream_end, usage_without_done, local_stop_after_text}` 时返回 true——
   这是流协议层的硬信号，与语言无关。
2. **cursor-relay-runner.js `hasIncompleteWorkAtEndByMode`**：去掉 `!upstreamError`
   一刀切守卫；上游报错时若 toolCalls/sawMutationTool/sawReadOnlyTool/incompleteTodos/
   streamIncomplete 任一为真，判 unfinished。无错误时若 streamIncomplete 也判 unfinished。
3. **js/mode/plan-mode.js `hasIncompleteWorkAtEnd`**：同上去掉 `if(upstreamError)return false`，
   用 `streamEndedWithoutReliableCompletion` 替代文本意图。
4. **cursor-relay-runner.js `isRecoverableStreamError`**：扩展匹配 HTTP 502/503/504、
   fetch failed、ECONNRESET/ECONNREFUSED/ETIMEDOUT/ENOTFOUND/EAI_AGAIN、socket hang up、
   connection reset/refused。**去掉**过于宽泛的 network/terminated/aborted/bad gateway 等
   自然语言词，只保留协议级错误码和我们自己生成的摘要前缀（上游服务异常/暂时无响应）。
5. **cursor-relay-runner.js 主循环可恢复错误分支**：补 `session.unfinishedWorkAtEnd = true`。
6. **cursor-relay-runner.js `failAgentRelaySession`**：检测本轮是否有工作进行中
   （toolResultSummaries 非空 / pendingToolCalls 非空 / incompleteTodos 非空）或错误可恢复，
   若是则标 unfinished 保留上下文。**去掉** `looksLikeIncompleteContinuationText(latestAssistantText)`
   文本检查，只保留硬信号。
7. **cursor-relay-runner.js `shouldContinueIncompleteWork`**：新增 `streamEndedWithoutReliableCompletion`
   触发续接——流没正常 done 就结束时，自动续接提示模型完成本轮，而非靠猜话术。

### 验证
- `node --check` 两文件语法通过；模块完整 require 加载无错误。
- 硬信号测试全通过：
  - looksLikeIncompleteContinuationText 只认 placeholder，中/英/日/韩/俄文继续意图全判 false。
  - streamEndedWithoutReliableCompletion：sawDone=true→false；stream_end/usage_without_done/
    local_stop_after_text→true；upstream_error→false。
  - isRecoverableStreamError：fetch failed/HTTP 502/ECONNRESET/timeout→true；400/generic→false。
- 修复后三种失败模式行为：
  - 5989b585（fetch failed）→ recoverable=true → unfinished 保留上下文。
  - ecd2955a（流没 done 就结束无 tool）→ streamIncomplete=true → 触发自动续接重提示模型。
    （若上游确实发了 done=true，则协议上算完成，relay 不应靠猜话术覆盖——这是正确的。）
  - 650cd896（4轮工具后 502）→ toolResultCount>0 + recoverable → unfinished 保留4轮工作。

## 速度优化：代理秒开关 + 响应缓存命中率提升

### 用户需求
1. 代理启动/关闭太慢，不能秒开关，重点查 Relay 启动拖速环节
2. 核心优化：大幅提升缓存命中率，本地对话数据全在 `C:\Users\Administrator\.cursorpool\relay\history`

### 代理启停瓶颈定位（cursor-relay-runner-manager.js / cursor-relay-proxy.js）
- `waitForRunnerHealth`：40次×150ms 固定间隔串行轮询，最坏6s
- 关闭流程：`requestRunnerShutdown(700)` + `waitForRunnerDown(8×100)` + `waitForRunnerDown(6×120)`，最坏2.2s
- `cursor-relay-proxy.js` 多处硬编码 sleep：quitCursor后 sleep(800)×3处、killForce后 sleep(1500)、reload后 sleep(1500)、resetConversation前 sleep(250)
- `ensureRelayCertificates`：4次 openssl execFileSync 串行（已有 fileReady 跳过，二次启动不触发）

### 启停优化（已实施）
1. **waitForRunnerHealth**（manager.js）：前8次用40ms短间隔+120ms超时快速探测，之后用配置间隔。进程1s内启动可~300ms检测到（原150ms×N）。
2. **关闭流程**（manager.js stopLocalRelayRunner）：requestRunnerShutdown 700→400ms，waitForRunnerDown 8×100→6×60 + 6×120→4×80。最坏2.2s→1.1s。
3. **sleep 优化**（proxy.js）：quitCursor后 800→300ms（3处）、killForce后 1500→500ms、reload后 1500→800ms、resetConversation前 250→80ms。restartCursor路径节省~1.8s。

### 缓存命中优化（核心价值，已实施）
**原状**：命中率0%——无任何请求复用层，每次都打上游。历史记录只用于会话状态追踪和记忆注入，流式SSE响应未做请求级缓存。

**新方案**：新建 `js/utils/relay-response-cache.js`，两层缓存策略：
- **exact 层**：`sha256(model + 完整messages + toolKey + phase)`，命中重试/重连的完全相同请求
- **fuzzy 层**：`sha256(model + 最后一条用户消息文本)`，仅对单轮请求（无assistant历史）查询，命中历史已回答过的相同问题

**安全约束**：
- 只缓存成功、有文本、无工具调用、无错误的响应
- 含 tool_calls/tool 角色的 messages 不生成 key（有副作用）
- 多轮对话不触发 fuzzy（避免上下文误命中）
- 只缓存 upstream/stream 主阶段，post_mutation_summary 等不缓存
- LRU 上限 2000 条，TTL 7天

**注入点**（cursor-relay-runner.js）：
- `fetchUpstreamCompletion` 入口：buildCacheKeys → get，命中返回伪 Response（body=null 触发 parseSseStream 的 text() 分支，catch 后 onDelta({text,done:true})）
- 成功响应：在 response 上标记 _cacheKeys
- `streamAgentUpstreamResponse` finally：仅当 !aborted && !upstreamError 时 set 缓存
- runner listen 后：`warmupFromHistory()` 异步从 history 目录扫描 user_message→assistant_text 对预热 fuzzy 缓存（setImmediate 不阻塞启动）

**验证结果**：
- 语法检查4文件全通过
- 单轮请求生成 exact+fuzzy，多轮只 exact，含tool_calls不缓存
- 历史预热：60个会话扫描，加载42条 fuzzy 缓存
- 历史"你是谁"fuzzy命中，秒回历史答案
- 伪 Response 正确触发 parseSseStream 回放分支

## 缓存策略升级：三层混合（exact → fuzzy → ultra）

### 用户决策
用户确认：同模型才命中是主路径（不串味），但同模型没命中时退回跨模型兜底（ultra）省 token。
即"混合：默认同模型，无命中时退回跨模型"。

### 三层结构（relay-response-cache.js 重写）
1. **exact** = `sha256(exact|model|toolKey|phase|完整messages)` —— 重试/重连相同请求
2. **fuzzy** = `sha256(fuzzy|model|最后用户消息)` —— 同模型对同一问题的历史回答（用户偏好）
3. **ultra** = `sha256(ultra|最后用户消息)` —— 跨模型兜底，最大限度省 token

查询顺序：exact → fuzzy → ultra（任一命中即返回，跳过上游）
写入：runtime 三层全写；历史预热看能否查到 model

### 历史预热的 model 来源问题
- `context.json` 的 items **完全没有 model 字段**（只 role/kind/payload/turn_seq/request_id）
- `state.json` 的 `last_provider_call` 是空对象 `{}`
- model 只在 `usage.db` 的 `relay_usage.model` 列里，按 `request_id` 关联
- 预热时 `_loadModelMapFromUsageDb()` 用 better-sqlite3 反查，按 request_id → model 填充 fuzzy 层
- **降级**：better-sqlite3 加载失败（外部 node 测试时）或 usage.db 不存在 → 只写 ultra 层
- 生产环境 runner 是 Electron fork 子进程，能正常加载 better-sqlite3

### 剔除策略：LRU + 频次保护（近似 W-TinyLFU）
- `MAX_MEMORY_ENTRIES = 3000`（从 2000 调到 3000，混合策略需要更多容量）
- `HOT_HIT_THRESHOLD = 3`：hits ≥ 3 视为热点
- `PROTECTED_HOT_ENTRIES = 500`：热点保护上限
- 剔除时：遍历找出非热点条目，一次剔除 10%（300条），跳过热点
- TTL 保持 7 天

### getStats 升级
返回 exactEntries/fuzzyEntries/ultraEntries/hotEntries/totalHits，供 UI 统计展示

### runner.js 配套改动
- `streamAgentUpstreamResponse` 写入缓存时带 `model: session.configuredModel || session.modelName`
- 这样 set 内部能把 model 写进 entry，预热反查时也能用到

## 上下文感知缓存（结合 deepseek 建议落地）

### Deepseek 三大缺陷评估
| # | 问题 | 真实性 | 处理 |
|---|------|--------|------|
| 1 | isSingleTurn 太严苛，多轮废 fuzzy/ultra | **真** | 改为 isCacheableConversation：最后是 user 且无副作用工具即可 |
| 2 | _set 重置 hits 破坏热点保护 | **真** | _set merge：保留旧 hits，刷新 ts |
| 3 | exact 缺 temperature/top_p | **半真** | 项目根本不传 temperature；但 reasoning_effort + disableReasoning 确实漏了，已补 |

### 工具分类（结合 cursor-relay-protocol.js getAgentNativeToolSpec）
- **幂等(IDEMPOTENT_TOOLS)**：read/ls/grep/glob/diagnostics/readlints/semanticsearch/webfetch/websearch/list_mcp_resources/read_mcp_resource → 可缓存
- **副作用(SIDE_EFFECT_TOOLS)**：shell/shell_stream/write/edit/strreplace/str_replace/delete/todowrite/createplan/task/computer_use/record_screen/write_shell_stdin/execute_hook/mcp → 不缓存
- **未知工具**：保守按副作用处理（hasSideEffectTools 中：不在幂等列表就视为副作用）

### 上下文指纹 buildContextFingerprint
- 单轮：`U:用户消息`
- 多轮(仅幂等工具)：`U:user1 \x00 T:tool_name:args \x00 R:tool_result \x00 U:user2`
- 跳过 system 和 assistant 文本（要缓存的内容）
- 工具结果截断 8192 字符保证稳定性
