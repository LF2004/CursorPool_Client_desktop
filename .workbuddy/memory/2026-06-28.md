# 2026-06-28 Native Multitask / Debug 模式原生任务 UI 修复

## 背景
按 `docs/native-multitask-debug-handoff-2026-06-28.md` 文档继续修复两个模式的剩余问题。文档指出：runner 控制流已通，但原生任务 UI 仍显示占位卡片（"New subagent" / "Planning next moves"），根因在 TaskStreamLog RPC 层与官方 proto schema 不对齐。

## 根因诊断（通过逆向 workbench 确认）
1. **proto schema 缺字段**：本地 `proto/regen/aiserver_v1.proto` 缺少官方 workbench 中存在的字段：
   - `TaskInfoResponse` 缺 `task_status`(2) enum 字段
   - `TaskStreamLogResponse` 缺 `info_update`(2) oneof 分支及其嵌套 `InfoUpdate` 消息
   - `InterfaceAgentStatus` 缺 8 个 `InterfaceAgentStepStatus` enum 字段(1-7,15)
2. **UI 卡片标题来源**：workbench 渲染层不直接读 `human_readable_title`，而是从 composer 名 / backgroundWorkTitle / placeholderTitle 解析；真正的 `human_readable_title` 消费发生在扩展宿主服务层（把 initial_task_info / info_update 写入 composer 状态）。所以必须正确发 initial_task_info + info_update 帧。
3. **streamedLogItem 内容丢失（既有 bug）**：`encodeTaskLogItemPayload` 返回已编码 Buffer，再被 `encodeTaskStreamLogFrame` 当作 JS 对象传给 `encodeCursorProtoMessageSync`，protobufjs 静默丢弃 oneof 内容，导致解码出空 `{}`。
4. **首条 log 类型错误**：任务首条 log 是 `thought` 而非 `instruction`，不符合官方任务日志顺序。
5. **TaskStreamLog 是一次性快照**：原来发完 initial + logs 就 close，没有 server-streaming 行为，无法推送 status 变更。

## 修复内容

### proto schema 对齐 (`proto/regen/aiserver_v1.proto`)
- `TaskInfoResponse` 增加 `TaskStatus task_status = 2`
- `TaskStreamLogResponse` 增加 `info_update = 2` oneof 分支 + 嵌套 `InfoUpdate { human_readable_title, task_status }`
- 新增 `InterfaceAgentStepStatus` enum (UNSPECIFIED/WAITING/RUNNING/SUCCESS/FAILURE)
- `InterfaceAgentStatus` 增加 8 个 enum 字段 (validate_configuration/stub_new_function/verify_spec/write_tests/write_implementation/implement_new_function/run_tests/write_test_plan)

### runner 核心 (`js/utils/cursor-relay-runner.js`)
- **新增 `handleTaskStreamLogLive`**：TaskStreamLog 改为真正的 server-streaming —— 发 initial_task_info（带 task_status）→ 增量推送 streamed_log_item → status 变化时发 info_update → 终态时关闭。30s 超时兜底，300ms 轮询。
- **修复 streamedLogItem 编码丢失**：拆出 `buildTaskLogItemObject`（返回 JS 对象），`encodeTaskStreamLogFrame(1, ...)` 直接消费对象，protobufjs 正确编码嵌套 oneof。
- `encodeTaskStreamLogFrame` 增加 responseField===2（info_update）分支。
- `encodeTaskLogItemPayload` 增加 `tool_action` / `tool_result` 两种 log_item 类型支持。
- 任务首条 log 从 `thought` 改为 `instruction`（用 description/prompt 作为 instruction 文本）。
- 子任务执行日志从 output/thought 改为 tool_action + tool_result，更贴近官方子代理生命周期。
- 导出纯函数（encodeTaskStreamLogFrame/buildTaskLogItemObject/handleTaskStreamLogLive 等）用于诊断与测试。

## Round 2 修复（同日续修 — 用户反馈截图后）

### 根因诊断（深入 proto 层）
用户截图显示：子 agent 卡片仍为 "Planning next moves" 占位符；debug 模式输出大量纯文本而非原生证据卡片。深入排查发现：

1. **backgroundSubagentAction / backgroundTaskCompletionAction 帧为空壳**：`InteractionUpdate` oneof **没有** `conversation_action` 字段（该字段属于 `AgentClientMessage` = 客户端→服务端方向）。`buildAgentBackgroundSubagentActionFrame` / `buildAgentBackgroundTaskCompletionActionFrame` 试图把 `conversationAction` 塞进 `InteractionUpdate`，protobufjs 静默丢弃 → 产生空帧。服务端永远不能通过交互流发送 `ConversationAction`。
2. **`taskToolCallDelta` 在 partial/started 阶段就带 result**：`buildStructuredToolCallSnapshot` 的 task 分支总是填充 `result.success`，即使任务还在运行中。客户端看到 success result 后认为任务已完成，不打开 `TaskStreamLog` → 子卡片无内容 → "Planning next moves"。
3. **子任务无法被 TaskStreamLog 找到**：`findTaskRecordAcrossSessions` 仅按 `taskUuid`(registry key) 查找，但客户端用 `taskToolCallDelta.callId`(= `parentToolCallId`) 发 `TaskStreamLogRequest.task_uuid`。两者不同 → 找不到子任务 → 回退到空全局任务。
4. **`subagent_type` 传值方式不对**：`emitTaskProgressFrame` 传 `{ [type]: type }` 对象给 `buildTaskSubagentTypeProto`，虽然 `normalizeTaskSubagentType` 能解析，但不够清晰。
5. **TaskError 字段名错误**：proto `TaskError` 有 `string error = 1`，代码用 `{ errorMessage: ... }` → 静默丢弃。
6. **ToolCall 缺 metadata**：`ToolCall` proto 有 `tool_call_id=57`/`started_at_ms=59`/`completed_at_ms=60`，代码未填充 → 客户端无法关联 delta 与 TaskStreamLog。
7. **debug 模式大段纯文本**：`getPostToolTurnAction` 返回 `buildLocalDebugConclusion()`(≤2200字) 作为 finalText → 全部输出为 textDelta；`ensureDebugArtifactsForTask` 向 task.log 追加合成的 "Debug Logs captured:" 等假证据条目。
8. **debug 超时太短**：`waitForBackgroundExecutions` debug 仅 2500ms，子代理跑 Read/ReadLints/Grep 不够 → `isDebugEvidenceMissing` 恒真 → `getPostToolTurnAction` 恒返回 null → 死循环超时。

### 修复内容

#### `js/utils/cursor-relay-runner.js`
- **`emitTaskProgressFrame` 重写**：
  - 删除 `buildAgentBackgroundSubagentActionFrame` / `buildAgentBackgroundTaskCompletionActionFrame` 调用（空壳帧，InteractionUpdate 无 conversationAction 字段）
  - 删除 `buildAgentToolCallProgressFrame` 重复帧（malformed ToolCallDeltaUpdate）
  - `subagent_type` 改传字符串（而非 `{ [type]: type }` 对象）
  - `execution.includeResult` = 仅终态(completed/failed)为 true
  - 新增 `startedAtMs` / `completedAtMs` 传给 ToolCall metadata
  - `resultSuffix` / `transcriptPath` 仅终态填充
- **`findTaskRecordAcrossSessions` 增加 `parentToolCallId` 别名查找**：遍历 registry 中所有 record，匹配 `record.parentToolCallId` 与请求 ID。session 级和 global 级都加。这样客户端用 callId 发 TaskStreamLog 也能找到子任务。
- **`ensureDebugArtifactsForTask` 重写**：仍填充 artifacts 容器（供 `isDebugEvidenceMissing` 检查）但 **不再向 task.log 追加合成证据条目**，避免污染 TaskStreamLog。
- **debug `waitForBackgroundExecutions` 超时 2500→8000ms**：debug 子代理跑 ReadLints/Diagnostics 需要更长时间。
- 导出 `emitTaskProgressFrame` / `findTaskRecordAcrossSessions` / `registerTaskSubagent` / `getSessionTaskRegistry` 供测试。

#### `js/utils/cursor-relay-protocol.js`
- **`buildStructuredToolCallSnapshot` task 分支重写**：
  - `execution.includeResult === false` 时不输出 `result` 字段（partial/started 阶段无 result）
  - `TaskError` 字段名 `errorMessage` → `error`（匹配 proto `TaskError.error = 1`）
- **`buildAgentTaskToolCallDeltaFrame` 增加 ToolCall metadata**：填充 `toolCallId`(field 57) / `startedAtMs`(field 59) / `completedAtMs`(field 60)。

#### `js/mode/debug-mode.js`
- **`getPostToolTurnAction` 重写**：不再返回 `buildLocalDebugConclusion()`(≤2200字大段文本)，改为返回简短标记（"Debug investigation complete. Review the subagent card for evidence and findings."）+ `markCompleted: true`。证据细节留在子代理 TaskStreamLog 的 tool_action/tool_result 条目中。
