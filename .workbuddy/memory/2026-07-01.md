## 技术细节

### buildCursorChecksum 函数（Go 实现，仅作参考）
`cursor-tab-server/main.go` 中的 `buildCursorChecksum` 函数：
1. 使用当前时间戳（毫秒除以 1_000_000）生成 6 字节 timestamp
2. 使用初始种子 165 对 timestamp 字节进行 XOR 和加法操作
3. 将结果编码为 base64（去除 `=` 填充）
4. 计算 Authorization header 的 SHA-256 哈希
5. 返回 `prefix + hash[:32]`

当前项目不需要这个函数，因为不是直接转发到 `api2.cursor.sh`。

## 2026-07-01 第二轮：修复 Build Plan 无响应（Task #14）

### 根因分析
当用户点击 "Build Plan" 按钮后：
1. Plan SSE 流已被 `finalizeWaitingInteractionSessionStream` 关闭（`streamDetached=true`, `res=null`）
2. 客户端发送 `conversation_action`(execute_plan_action) → `scheduleExecutePlanSessionResume` 设置 300ms 定时器
3. 300ms 定时器因 `streamDetached || !session.res` 跳过（设计如此）
4. 客户端发送新 RunSSE 连接
5. `handleAgentRunSse` 先尝试 `agentSessions.get(requestId)`（path 1）— 若 RunSSE 使用不同 requestId 则失败
6. 再尝试 `findWaitingSessionByStableConversationId`（path 2）— 需要 `pendingCapture` 中的 `stableConversationId`
7. `pendingCapture` 来自 `pendingAgentMessages.get(requestId)`，但 `pendingAgentMessages` 只在 `user_message` BidiAppend 时填充
8. `conversation_action` 不填充 `pendingAgentMessages` → `pendingCapture=null` → `stableConversationId` 为空
9. path 2 也失败 → 创建新 session → Plan 执行不触发

### 修复内容

**文件：`js/utils/cursor-relay-runner.js`**

在 `handleAgentRunSse` 的 path 1 和 path 2 都失败后、创建新 session 之前，添加兜底逻辑：

```javascript
// Fallback: search all sessions for one awaiting runsse rebind
if (requestId) {
  for (const candidate of agentSessions.values()) {
    if (candidate && !candidate.aborted && !candidate.completed
        && candidate.awaitingRunsseRebind === true && !candidate.relaying) {
      rebindWaitingSessionRequestId(candidate, requestId, logger);
      reattachWaitingInteractionSession(candidate, req, res, rawBody, logger);
      triggerDeferredInteractionResume(candidate, config, logger, stats, 'runsse_awaiting_fallback');
      return;
    }
  }
}
```

该兜底逻辑遍历所有活跃 sessions，找到标记了 `awaitingRunsseRebind === true` 的 session 并将新 RunSSE 绑定上去。

## 2026-07-01 第三轮：修复"启用 Relay 失败 — Missing named parameter 'completion_model'"

### 根因
`saveRelayProfileStore` 的 SQL `INSERT` 语句包含了 `completion_model` 列和 `@completion_model` 占位符，但 `insert.run()` 调用时没有传入 `completion_model` 字段，导致 better-sqlite3 报 `RangeError: Missing named parameter "completion_model"`。

同样的问题存在于三处：
1. `normalizeStore` — 归一化时未映射 `completionModel`
2. `loadRelayProfileStore` — 加载时未读取 `completion_model` 列到返回对象
3. `saveRelayProfileStore` — 保存时未传入 `completion_model` 参数

### 修复
在 `js/utils/cursor-relay-profile-store.js` 中补全三处 `completionModel`/`completion_model` 映射：
- `normalizeStore` L198: 添加 `completionModel: String(item?.completionModel || '').trim()`
- `loadRelayProfileStore` L285: 添加 `completionModel: String(row.completion_model || '')`
- `saveRelayProfileStore` L347: 添加 `completion_model: item.completionModel || ''`

## 2026-07-01 第五次：修复对话卡死 Reconnecting... / write_failed 崩溃

### 问题现象
用户反馈：
1. 对话很慢（50秒无响应）
2. UI 显示 "Reconnecting..." 卡死
3. runner 日志显示 `session aborted reason=write_failed:Cannot read properties of null (reading 'write')`

### 根因分析（从 runner.log 逐条分析）
- `03:29:11` RunSSE 打开 → `03:29:12` BidiAppend 用户消息 "你好啊"
- `03:29:12` 开始 upstream fetch (mode=chat)，超时 300s，代理 `windows_system:http://127.0.0.1:7897`
- `03:30:03` `bridge CONNECT failed api2.cursor.sh:443: read ECONNRESET` — 代理层网络断开
- `03:30:26` chat 模式完全失败 `Client network socket disconnected before secure TLS connection was established`
- `03:30:26` fallback 到 responses 模式，再次 upstream fetch
- `03:30:27` 客户端因超时关闭 RunSSE → `runsse_closed` → session detached → `res` 被设为 null
- `03:31:06` responses 上游终于返回 200，但只返回了 `response.completed` 空内容（textLen=0）
- 代码尝试 `session.res.write(buffer)`，但 `res` 已在 detached 时被设为 null
- 抛出 `TypeError: Cannot read properties of null (reading 'write')`
- `writeAgentFrame` 的 catch 块调用 `abortAgentSession(..., 'write_failed:...')`
- session 被完全 abort，客户端后续重连的 BidiAppend 被标记为 `duplicate user_message ignored`
- 客户端只能反复发 heartbeat，永远等不到响应，UI 卡死在 Reconnecting...

### 修复
`js/utils/cursor-relay-runner.js` 的 `writeAgentFrame` 函数：
- 原逻辑：只检查 `session?.active`，不检查 `session.res` 是否为 null
- `shouldBufferAgentFrameWhileDetached` 只在 `streamDetached === true && !session.res` 时缓冲
- 但如果 session 因其他原因（如 runsse_closed）detached，`streamDetached` 可能还没被设为 true，导致 `shouldBufferAgentFrameWhileDetached` 返回 false，继续执行 `session.res.write()`
- 修复：在 `shouldBufferAgentFrameWhileDetached` 之后、执行 `res.write` 之前，添加显式 `if (!session.res) return;` 检查，避免 null pointer

### 结论
- "很慢" 主要是上游网络问题（socket hang up / ECONNRESET）
- "卡死" 是 runner 代码 bug：res null 时 write 崩溃并 abort session
- 修复后，即使网络不稳定导致 RunSSE 提前关闭，runner 不会 crash，会优雅地丢弃无法发送的帧
